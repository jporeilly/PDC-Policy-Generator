How this app fits the workflow — and why the Registry exists

In PDC, the same three facts about a column — which business term it maps to, what tags it carries, and how sensitive it is — get decided in more than one place. The Glossary Generator owns the reviewed vocabulary; at export it writes a Classification Registry: one row per governed concept carrying the term, the governed tags, the sensitivity, and the detection seeds its scan induced from profiled data (value-format regexes like ^CSCU-\d{6}$, profiled reference-value lists). This app reads that Registry and authors PDC's Data Identification methods from it — so the term a method binds and the tags it stamps can never quietly diverge from the glossary. Everything runs locally and deterministically: no LLM, no network — the evidence was already gathered at scan time.

Authorturn each detection seed into an import-ready method (this page — working) Reconcilebind each method to the term's minted PDC term_id once Resolve has backfilled the Registry (roadmap) Deployimport methods over the public API (v3) and trigger the DATA_IDENTIFICATION jobs (roadmap) Drift-checkcompare deployed Assign-Tags + PDC's live tag facet against the Registry's governed vocabulary (roadmap)

What the summary numbers mean
Conceptskept, steward-reviewed terms in the exported glossary — one Registry row each With seedsconcepts with evidence to author from: an induced value-format regex ({type:"pattern"}) or a profiled reference-value list ({type:"dictionary"}). Only these become methods — guessing from names is exactly what this pipeline avoids. Resolved term idsconcepts whose term_id PDC has minted. 0 is normal before glossary import: import the glossary, run the Glossary app's Resolve term ids, re-export — until then rules bind the term by name, which PDC accepts. Governed tagsthe embedded tag_vocabulary.allow_list — the same list that drives the Glossary app's tagging. One vocabulary, two apps. Off-vocabularytags that escaped the allow-list on the glossary side. Fix them there (Dictionary page) — authoring refuses them rather than letting a stray facet into PDC.

Loading is read-only and local: the file is parsed, the classification-registry/1 envelope validated, nothing is written and nothing leaves your machine. The full contract is documented field-by-field in docs/CONTRACT.md.

Every emitted file is the exact shape PDC's Management → Data Identification → Import accepts — the same fields the CSCU Technical Track (Module 03) teaches, so a steward can read every rule.

nameprefixed (e.g. CSCU Member Number) so your methods group together in PDC's method list column hinta column-name regex derived from the Registry's physical sources[] — the real columns the steward mapped, not a guess from the term contentPatternsthe induced value signature (e.g. ^CSCU-\d{6}$), built from profiled sample values at scan time (position signatures, ≥90% share) dictionary CSVthe reference values profiling actually saw (risk ratings, SAR statuses, account types) — similarity-weighted matching in PDC applyTagsthe concept's governed tags, re-filtered against the allow-list at authoring time — always lower-case (tags are search facets; case variants would fragment them) assignBusinessTermthe governed term — by term_id when resolved, by name until then INDEX.csvthe review manifest: kind, rule name, file, term, term_id — read it before importing anything

Unzip, import the methods, run Data Identification on the sources, verify the governed tags landed. The CSCU workshop (PDC-Scenarios repo, courseware/CSCU/Policy/) walks the whole flow.

Under the hood — what happens in PDC (manual today, the deploy stage tomorrow)

Today you import through the PDC UI — the zip is shaped for exactly that. The deploy stage will run the same lifecycle over the public API (v3), following the Glossary app's proven adapter (Bearer auth; individual job endpoint first, bulk /jobs/execute/bulk fallback on 404/405).
